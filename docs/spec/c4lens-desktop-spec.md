# c4lens Desktop (Tauri) Implementation Spec

| | |
|---|---|
| **Status** | Implementation-ready draft |
| **Date** | 2026-06-28 |
| **Owner** | fuentesjr |
| **Design basis** | [`docs/design/c4lens-desktop-design.md`](../design/c4lens-desktop-design.md) |
| **Deliverable** | Desktop/Tauri implementation contract for Phases 0-2 |

This spec turns the desktop design into build instructions. The design document
records the product reasoning; this document records the contracts a builder
must implement: file formats, schema, validation, merge behavior, IPC, SQLite
DDL, generation algorithms, module boundaries, workflows, and acceptance
criteria.

The desktop/Tauri direction is chosen. Do not re-litigate earlier platform
alternatives while implementing this spec. Any prior exploration that affects
the builder contract is intentionally restated here, so this spec can be
implemented without depending on moved or deleted design files.

## 1. Scope

### 1.1 MVP Scope

MVP is Phases 0-2:

- Phase 0: Tauri 2 shell, React/Vite renderer, xyflow canvas, hard-coded sample
  model, folder open flow, early WebView smoke checks.
- Phase 1: model parsing, JSON Schema validation, semantic validation,
  effective model loading, `ViewDeriver`, navigator, watcher, SQLite code
  index, jump-to-code, inline source preview, `c4lens validate`.
- Phase 2: code-to-model generation into `c4/model.generated.yml`, overlay
  merge, provenance badges, review-diff UX, search, dependency highlighting,
  SVG/PDF/PNG export, layout caching, macOS packaging, `c4lens generate`.

The target operating system for MVP is macOS only. Produce universal arm64 +
x86_64 bundles. Windows/Linux remain architectural constraints but are not
release blockers for MVP.

Current scaffold note: the initial Phase 0 implementation wires the Tauri
shell, folder-open flow, hard-coded sample model, renderer view derivation, ELK
layout, and xyflow canvas. The broader command surface, validation pipeline,
watcher, SQLite index, scanner, generator, and generation review workflow remain
Phase 1-2 contracts until their sections are implemented.

### 1.2 Non-Goals

- No drag-to-draw or drag-to-edit diagram canvas.
- No account, sync, cloud, teams, or multi-user collaboration.
- No multi-repo workspace.
- No rendered C4 Level 4 code diagrams in MVP.
- No local MCP server or live write API in MVP.
- No precise runtime/call-graph relationship inference in MVP.
- No generated-slug rename/move preservation in MVP.
- No tombstones or per-field overlay overrides in MVP.

### 1.3 Build-Time Gate Before App Scaffolding

Before scaffolding the Tauri shell, validate the generation approach with a
throwaway spike against one real codebase. The spike must not become production
code. It must answer:

- Which manifests are detected as containers?
- Which directories/modules become useful components?
- Which import/dependency relationships are noisy versus useful?
- Are path-derived generated slugs stable enough for the selected codebase?
- Does the overlay merge avoid clobbering authored content?

Proceed to the Tauri shell only after the spike yields one useful generated
`c4/model.generated.yml` draft and a short list of known generation limits.

## 2. Product Invariants

These are fixed implementation invariants:

- The architecture model is data, not a drawing.
- `c4/model.yml` is the authored source of truth.
- `c4/model.generated.yml` is generated, disposable overlay data.
- SQLite is a cache/index only; it is never canonical.
- Views are derived from the model at runtime; no stored coordinates.
- Layout is automatic through ELK in the renderer.
- The renderer owns navigation, view derivation, layout, and canvas state.
- The Rust core owns filesystem access, parsing, validation, scanning,
  generation, indexing, watching, and native integrations.
- Renderer/core communication uses Tauri IPC commands/events, not HTTP.
- Relationships may be declared at any C4 level and are aggregated to the
  relevant boundary by `ViewDeriver`.

## 3. Repository Files

### 3.1 Model Files

Within an opened repository's C4 model directory, the app reads and writes only
these model files:

```text
c4/
  model.yml            # authored by humans/agents; source of truth
  model.generated.yml  # generated by c4lens; safe to overwrite
  schema.json          # JSON Schema for both model files
```

`model.yml` may be YAML or JSON by content, but the default filename is YAML.
`model.generated.yml` is always emitted as YAML.

### 3.2 Missing File Behavior

- Missing `c4/` directory: show an empty repository state with actions to scan
  and generate. `open_repo` still succeeds.
- Applying generated output in a repo with no `c4/` directory creates a real
  `c4/` directory under the canonical repo root, writes the bundled
  `schema.json` if absent, and atomically writes `model.generated.yml`. It does
  not create `model.yml`.
- Missing `c4/model.yml`: effective model is generated-only if
  `model.generated.yml` exists; otherwise no model is loaded.
- Missing `c4/model.generated.yml`: effective model is authored-only if
  `model.yml` exists.
- Missing `c4/schema.json`: app validation uses the bundled schema; `generate`
  writes the current bundled schema to `c4/schema.json` if the file is absent.
- Present but stale or edited `c4/schema.json`: app validation still uses the
  bundled schema. The repo schema is treated as an editor artifact; schema drift
  is reported as a warning and can be fixed by rewriting `c4/schema.json`.
- Invalid authored or generated model: keep the last valid effective model in
  the renderer and emit/show the validation report.

### 3.3 Path Rules

All paths stored in model files are repo-relative POSIX-style paths using `/`,
even on Windows later. IPC payloads may include absolute paths only for
OS/editor integration fields such as `RepoHandle.rootPath` and
`CodeRef.absolutePath`. Absolute paths must not be written to `model.yml` or
`model.generated.yml`.

`code` may point to a file or directory. If it points to a directory, jump to
code opens the directory and inline preview shows the best indexed source file
for the element when one exists.

All filesystem operations must enforce the repo boundary after canonicalization:

- Canonicalize the selected repo root once when opening the repo.
- For every model `code` path, source preview path, editor-open path, scan path,
  export source path, and indexed file path, join it to the repo root and
  canonicalize the resolved path before reading or opening it.
- For control files (`c4/model.yml`, `c4/model.generated.yml`, and
  `c4/schema.json`), join the fixed repo-relative path to the repo root and
  canonicalize the existing file or nearest existing parent before reading or
  writing. Reject the operation if an existing `c4/` directory, control file, or
  symlink target resolves outside the canonical repo root. Files outside the
  repo must remain unchanged.
- Reject the operation if the canonical resolved path is outside the canonical
  repo root.
- Do not follow symlinked directories during scans. File symlinks are allowed
  only when their canonical target remains inside the canonical repo root.

Path classification:

- Reject absolute paths, Windows drive paths, URL-like paths, paths containing
  backslashes, empty segments, NUL bytes, or other non-POSIX syntax as
  `semantic.code_path_invalid` for model `code` fields and `path.invalid` for
  IPC inputs.
- Normalize `.` segments before joining to the repo root.
- Reject paths whose normalized or canonical result escapes the repo root as
  `semantic.code_path_outside_repo` for model `code` fields and
  `repo.path_denied` for IPC inputs.
- Treat valid repo-relative paths that resolve inside the repo but do not exist
  as warnings (`semantic.code_path_missing`) in model validation.
- `open_in_editor` must perform the same classification and boundary checks
  before invoking the OS/editor opener.

## 4. Domain Model

### 4.1 Slugs

The map key is the element slug. Slugs are global across actors, systems,
containers, and components.

Slug grammar:

```text
^[a-z][a-z0-9_]*$
```

Generation slug rules:

- Lowercase path or manifest name.
- Replace non-alphanumeric runs with `_`.
- Trim leading/trailing `_`.
- If the result does not start with a letter, prefix `x_`.
- If a generated slug collides with an existing generated slug, append `_2`,
  `_3`, etc.
- If a generated slug collides with an authored slug, reuse that slug only when
  the generated element intentionally targets the same element identity: same
  slug, same type, and same parent slug. Otherwise append `_2`, `_3`, etc. until
  the generated slug is globally unique across authored and generated elements.

Generated slugs are path-derived. File moves are represented as delete+add.
Dangling authored relationships caused by moves are validation errors. Rename
or move detection is post-MVP.

### 4.2 Element Types

```text
actor      # human user, role, persona, or external participant
system     # software system; may be internal or external
container  # separately runnable/deployable unit or data store inside a system
component  # internal building block inside a container
```

Allowed hierarchy:

```text
model
  actors: actor*
  systems: system*
    containers: container*
      components: component*
```

External systems must not define containers in MVP. Treat that as a validation
warning, not an error, so imported/generated models can still be inspected.

### 4.3 Lifecycle

Allowed status values:

```text
live        # default
planned
deprecated
```

### 4.4 Container Kind

Allowed `kind` values:

```text
service  # default
app
store
queue
worker
job
```

Rendering hints:

- `store` renders as a data-store/cylinder-style card.
- `queue` renders as a queue/topic-style card.
- Other values render as normal rectangular C4 cards.

### 4.5 Relationship Identity

Relationships are directed edges. The normalized relationship identity key is:

```text
from + "\0" + to + "\0" + description + "\0" + (tech || "") + "\0" + (status || "live")
```

This key is used for de-duplication after shorthand normalization and overlay
merge. Authored and generated relationships with the same identity collapse to
one effective relationship, with `generated: false`.

```yaml
        name: API Application

        code: app/

          signin:
            name: Sign In Controller
            tech: Rails controller

            name: Accounts Controller

  - api -> mainframe: Calls (XML/HTTPS)
  - api -> email: Sends mail via
  - signin -> security: Uses
  - accounts -> mainframe: Uses
```

### 5.2 Relationship Forms

Unquoted shorthand form (preferred for humans):

```yaml
- from_slug -> to_slug: Description
```

YAML parses the unquoted shorthand as a one-entry mapping. c4lens treats that
mapping as shorthand, not as arbitrary relationship object syntax.

Quoted string shorthand form:

```yaml
- "from_slug -> to_slug: Description"
```

Object form:

```yaml
- from: api
  to: mainframe
  description: Calls
  tech: XML/HTTPS
  status: live
  generated: true
```

Parser normalization converts either shorthand form to object form before
semantic validation:

```json
{
  "from": "from_slug",
  "to": "to_slug",
  "description": "Description",
  "status": "live",
  "generated": false
}
```

If shorthand description contains a parenthesized suffix, keep the full text in
`description`; do not try to split technology out of shorthand.

### 5.3 `tech` and `technology`

`tech` is the canonical field in normalized models and IPC payloads.
`technology` is accepted in YAML as a compatibility alias. During
normalization:

- if only `technology` is present, copy it to `tech`;
- if both are present with the same value, keep one `tech`;
- if both are present with different values, emit
  `semantic.conflicting_technology_alias` and do not load the new model.

Generated YAML must write `tech`, never `technology`.

### 5.4 Unsupported YAML Features

MVP accepts plain YAML mappings, sequences, and scalars only. It must reject
anchors, aliases, and merge keys (`<<`) before schema validation.

Behavior:

- Do not expand aliases into the parsed model.
- Emit `parse.unsupported_yaml_feature` with line/column when the parser
  provides it.
- Treat the file as invalid for both GUI loading and CLI validation.

change `$id` or formatting, but not the accepted data shape without updating
this spec.

## 8. Validation Pipeline

The Rust core implements one validation pipeline shared by:

- Tauri commands.
- File watcher.
- CLI commands.
- Generation before write/apply.

Pipeline:

1. Read `model.generated.yml` if present.
2. Read `model.yml` if present.
3. Parse YAML/JSON into generic values with duplicate-key detection enabled.
   Duplicate keys are parse errors (`parse.duplicate_key`) with the duplicated
   YAML/JSON path when available. YAML anchors, aliases, and merge keys are
   rejected as parse errors (`parse.unsupported_yaml_feature`); the parser must
   not expand aliases into the model.
4. Normalize relationship shorthand in both sources.
5. Validate each source against the bundled JSON Schema.
6. If the authored source has parse/schema errors, return a failed report; do
   not merge. If only `model.generated.yml` has parse/schema errors and
   `model.yml` is valid, GUI model loading may ignore the generated overlay,
   load authored-only, and include warning
   `model.generated_overlay_invalid_ignored`. CLI `validate` with generated
   validation enabled still returns a failed report.
7. Normalize aliases and defaults: `technology` -> `tech`, `status: live`,
   `external: false`, `kind: service`, `generated: false`, and empty element /
   relationship collections.
8. Merge generated under authored to produce the effective model.
9. Flatten all elements into `elementsBySlug`.
10. Run semantic checks against the effective model.
11. Compute `sourceSha` from the bytes of both input files plus the bundled
    schema version.
12. Return `EffectiveModel` when there are no semantic errors.

Warnings never prevent rendering. Errors prevent the new model from replacing
the last valid renderer model.

Generated overlay recovery:

- GUI `get_model` loads authored-only when `model.yml` is valid and
  `model.generated.yml` is invalid.
- The UI must show the generated-overlay validation errors and offer actions to
  ignore for now, regenerate, or reveal the file.
- The app must not delete or rewrite an invalid generated overlay without an
  explicit user or CLI action.
- Generated-only repositories cannot recover this way; if
  `model.generated.yml` is the only model and is invalid, loading fails.

Semantic checks:

- Every slug matches the slug grammar.
- Every slug is globally unique after flattening the effective model.
- Every relationship source resolves.
- Every relationship target resolves.
- Actors and systems are top-level only.
- Containers appear only under systems.
- Components appear only under containers.
- External systems with containers produce a warning.
- `code` paths use supported repo-relative POSIX syntax.
- `code` paths do not escape the repo root after normalization and
  canonicalization.
- Missing but otherwise valid `code` paths inside the repo produce warnings,
  not errors.
- Conflicting `tech` and `technology` aliases are errors.

## 9. Overlay Merge

### 9.1 Effective Model

```text
effective = merge(generated, authored)
```

If both files are absent, there is no effective model.

If only one file exists, it is normalized and used as the effective model.

Top-level model fields merge before elements:

- `name` comes from authored `model.yml` when present, otherwise from
  `model.generated.yml`.
- `description` comes from authored `model.yml` when present, otherwise from
  `model.generated.yml` when present.
- `generated` is false for the effective model when authored `model.yml` exists,
  otherwise true for generated-only models.

### 9.2 Element Merge

Element identity is the global slug.

Rules:

- Start with all generated elements.
- Add all authored elements.
- If an authored element has the same slug as a generated element, the authored
  element's own fields replace the generated element's own fields as a unit.
- Child maps merge recursively by slug. If an authored system replaces a
  generated system, generated containers remain unless a container with the same
  slug is also authored. If an authored container replaces a generated container,
  generated components remain unless a component with the same slug is also
  authored.
- Replacement is whole-element for the element's own fields; there is no
  per-field merge of `name`, `description`, `tech`, `status`, `code`,
  `external`, or `kind`.
- `generated` is false in the effective element when authored wins.
- There are no tombstones in MVP.

This intentionally biases toward non-destructive author control over preserving
generated detail.

### 9.3 Relationship Merge

Rules:

- Normalize shorthand relationships to object form first.
- Start with generated relationships.
- Append authored relationships.
- De-duplicate by the relationship identity key in section 4.5.
- If authored and generated relationships share identity, keep one effective
  relationship with `generated: false`.
- If two generated relationships share identity, keep the first stable sort
  order occurrence.
- If two authored relationships share identity, keep the first and emit warning
  `semantic.duplicate_relationship`.

Relationships are additive. An authored relationship does not replace all
generated relationships for an element.

### 9.4 Stable Ordering

Generated overlay output must use deterministic ordering:

1. Actors by slug.
2. Systems by slug.
3. Containers by slug within each system.
4. Components by slug within each container.
5. Relationships by `from`, then `to`, then `description`, then `tech`.

## 10. ViewDeriver

The renderer implements `ViewDeriver`. It must be pure and deterministic:

```ts
deriveView(model: EffectiveModel, scope: ViewScope): DerivedView
```

No IPC calls are allowed inside `deriveView`.

If `scope.slug` is present but does not resolve to the element type required by
the requested level, `deriveView` returns an empty `DerivedView` with the
requested scope and no nodes or edges. The renderer is responsible for showing
the route as not found rather than crashing or silently changing scope.

### 10.1 Ancestor Helpers

```text
owner_at(element, level):
  if level == context:
    actor -> actor
    system -> system
    container -> parent system
    component -> parent container -> parent system

  if level == container:
    actor -> actor
    system -> system
    container -> container
    component -> parent container

  if level == component:
    actor -> actor
    system -> system
    container -> container
    component -> component
```

### 10.2 Context View

Scope:

```ts
{ level: "context", slug?: systemSlug }
```

If `slug` is absent, nodes are all actors and systems.

If `slug` is a system, nodes are:

- the scoped system;
- actors directly or indirectly related to it;
- systems directly or indirectly related to it.

For context view, "directly or indirectly related" means a relationship whose
endpoints project to the scoped system and the other actor/system. It includes
relationships from/to lower-level descendants of the scoped system after
projection, but it does not include graph transitive closure through unrelated
intermediate systems.

Edges are all relationships projected to actor/system boundaries. A relationship
between lower-level descendants is surfaced as a relationship between their
owning systems. Self-edges created by projection are discarded.

### 10.3 Container View

Scope:

```ts
{ level: "container", slug: systemSlug }
```

Nodes are:

- all containers inside the scoped system;
- actors/systems outside the scoped system that interact with the scoped
  system or its containers/components;
- external systems referenced by lower-level relationships.

Edges are relationships projected to container boundary for elements inside the
scoped system and actor/system boundary for elements outside it. Self-edges
created by projection are discarded.

### 10.4 Component View

Scope:

```ts
{ level: "component", slug: containerSlug }
```

Nodes are:

- all components inside the scoped container;
- other containers that scoped components communicate with;
- external systems/actors that scoped components communicate with.

Edges are relationships projected to component boundary for elements inside the
scoped container and nearest visible boundary for elements outside it.
Self-edges created by projection are discarded.

### 10.5 Edge Aggregation

Pseudocode:

```text
derive_edges(scope, visible_nodes):
  groups = ordered map

  for relationship in model.relationships:
    source = elementsBySlug[relationship.from]
    target = elementsBySlug[relationship.to]

    projected_source = project(source, scope)
    projected_target = project(target, scope)

    if projected_source is null or projected_target is null:
      continue

    if projected_source == projected_target:
      continue

    if either endpoint is not in visible_nodes:
      continue

    key = projected_source + "\0" + projected_target
    groups[key].relationships.append(relationship)

  for group in groups:
    sort group.relationships by relationship identity key
    emitted edge.generated = all grouped relationships are generated
```

Projected edge labels:

- One relationship: use its description.
- Multiple relationships with the same description: show it once.
- Multiple distinct descriptions: renderer shows a count on the edge and lists
  details in the side panel.

## 11. Layout

ELK runs in the renderer. Layout input is `DerivedView`.

Layout cache key:

```text
sourceSha + ":" + scope.level + ":" + (scope.slug || "root") + ":" + layoutInputSha
```

`layoutInputSha` is a stable digest of all inputs that can change ELK output:
derived node ids, edge ids, renderer layout version, layout options, measured
node dimensions, node badges/expanded states that affect size, font metrics, and
theme density. Cosmetic color-only theme changes do not need to invalidate
layout.

Cache invalidation:

- Any `sourceSha` change invalidates all layout entries.
- Any `layoutInputSha` change invalidates the matching scope entry.
- Renderer-only navigation does not invalidate layout.
- A hard reload clears in-memory layout cache.

ELK defaults:

```text
algorithm: layered
direction: RIGHT
spacing.nodeNode: 80
spacing.edgeNode: 40
spacing.edgeEdge: 20
layered.spacing.nodeNodeBetweenLayers: 120
```

Renderer may tune these values, but must keep deterministic layout for the same
input.

### 12.2 Commands

MVP supports one active repo per app window. Commands operate on the active repo
for that window unless they open or replace it. Opening a different repo stops
the previous watcher, closes its index connection, clears generation candidates,
and then installs the new repo as active. Events include `repoId`; renderers
must ignore events whose `repoId` does not match their active repo.

```ts
open_repo(params: { path?: string }): Promise<RepoHandle>
```

If `path` is absent, open a native folder picker. Store the chosen repo in app
state, initialize/open its SQLite index, and start file watching.

```ts
get_model(): Promise<EffectiveModel>
```

Load, validate, merge, and return the current effective model for the open repo.

```ts
validate_model(params?: { includeGenerated?: boolean }): Promise<ValidationReport>
```

Validate `model.yml` and, by default, `model.generated.yml` if present.

```ts
scan_codebase(params?: ScanOptions): Promise<ScanSummary>
```

Build or refresh the SQLite index. Emits `scan-progress`.

```ts
generate_model(params?: GenerateOptions): Promise<GenerationDiff>
```

Generate overlay YAML from the current index. If `scanFirst` is true, run a scan
first. This command never writes `c4/model.generated.yml`; it returns a
reviewable candidate identified by `candidateId`. The candidate is valid until
the repo changes, a new generation candidate is created for the repo, or the app
exits. The response records the authored model hash, overlay hash, effective
model source hash, index scan token, and bundled schema version used to create
the candidate.

```ts
apply_generated(params: ApplyGeneratedParams): Promise<void>
```

Apply requires all preconditions in `ApplyGeneratedParams` to match current
state. If the candidate is missing, fail with `generation.candidate_not_found`.
If the authored model hash, effective model source hash, index scan token, or
schema version differs from the candidate, fail with
`generation.candidate_stale`. If the overlay hash differs, fail with
`generation.overlay_changed`.

Apply is atomic:

1. Render the accepted candidate YAML in memory.
2. Acquire the repo write lock.
3. Create a real `c4/` directory under the canonical repo root if it is absent.
4. Write the candidate overlay to a temp file beside `c4/model.generated.yml`.
5. Validate the effective model using current `model.yml`, the temp overlay,
   current index preconditions, and the bundled schema.
6. If validation has errors, delete the temp overlay and fail without touching
   existing model/control file contents.
7. If `c4/schema.json` is absent, write the bundled schema to a temp file beside
   it and rename it into place. If this editor-artifact write fails, delete the
   temp overlay and fail with `fs.write_failed`; runtime validation never relied
   on the repo schema.
8. Rename the temp overlay over `c4/model.generated.yml`.
9. Release the repo write lock.
10. Emit `model-changed`.

The overlay replacement is all-or-nothing. A newly created empty `c4/` directory
may remain after failure, but failed validation must not replace
`model.generated.yml` or create a misleading partial overlay.

MVP may implement only `acceptAll: true`. If per-change selection is not
implemented, the UI must not present per-change apply controls.

```ts
get_element_code(params: { slug: Slug }): Promise<CodeRef | null>
```

Resolve code by explicit model `code`, then `element_sources`, then best indexed
symbol/file. Return null if there is no usable code reference.

```ts
open_in_editor(params: { path: string; line?: number; column?: number }): Promise<void>
```

Open a repo-relative path in the configured editor or OS default. The command
must reject invalid path syntax before invoking the opener and must reject
paths outside the repo root.

```ts
search(params: { query: string; limit?: number }): Promise<SearchResults>
```

Search elements, indexed files, and indexed symbols. Default limit is 20 per
category.

Search semantics:

- Trim leading/trailing whitespace.
- Maximum query length is 256 Unicode scalar values after trimming; longer
  queries fail with `search.query_invalid`.
- Empty queries return empty result arrays.
- Matching is case-insensitive substring matching for element slug/name/
  description/tech, file path, symbol name, and symbol qualified name.
- SQL searches must use prepared statements; wildcard characters in the user
  query are treated as literal text.
- Results are deterministic: elements sort by slug, files by path, and symbols
  by path, then range start, then name.

```ts
export_view(params: {
  format: "svg" | "pdf" | "png";
  scope: ViewScope;
  svg?: string;
  pdfBase64?: string;
  pngBase64?: string;
}): Promise<{ savedPath: string }>
```

Renderer supplies `svg` for SVG export, `pdfBase64` for PDF export, and
`pngBase64` for PNG export. Core opens a native save dialog and writes the
selected file. Phase 2 is complete when SVG, PDF, and PNG save from the app.

### 12.4 Repo Write Lock

Any operation that writes `c4/model.generated.yml`, `c4/schema.json`, the SQLite
index, or `model_cache` must hold an exclusive repo write lock. The lock is
keyed by `repoId` and must coordinate GUI and CLI processes, for example via an
advisory lock file under the c4lens app support directory.

If the lock is already held:

- GUI commands fail fast with `repo.write_locked`.
- CLI commands exit with code 3 and print a concise diagnostic.
- Watcher-triggered work is deferred and retried after debounce instead of
  surfacing a user error.

The lock must be released on success or failure. Partial temp files from failed
overlay writes are removed before returning the error.

### 13.2 Incremental Rescan

Algorithm:

```text
scan(repo):
  previous = load files by path
  current = walk repo with ignore rules

  for each current file:
    stat file
    apply parser eligibility filters
    compute content_sha
    if previous[path].content_sha == content_sha:
      update mtime/size only
    else:
      delete element_sources rows for the changed file
      delete file's symbols/imports
      parse file if language supported
      insert symbols/imports
      upsert file row

  for each previous path not in current:
    delete file row cascade

  update repos.scanned_at/head_sha/scan_token
  delete model_cache for repo if scan changed any indexed file
```

A full rescan deletes all `files`, `symbols`, `imports`, and `element_sources`
rows for the repo before rebuilding. Incremental scan deletes and rebuilds
`element_sources` for changed or deleted files so jump-to-code and generation
never use stale source mappings.

`mtime_ms` and `size_bytes` are advisory metadata only. They must not be the
sole reason to skip hashing a file, because same-mtime/same-size edits must
still update `content_sha`, symbols, imports, and `scan_token`.

`scan_token` is a stable digest of indexed file paths plus content hashes after
the scan completes. It changes whenever generation-relevant indexed content
changes, even if the VCS `head_sha` is unavailable or unchanged.

The scanner must respect `.gitignore` through the Rust `ignore` crate. Always
exclude:

```text
.git/
node_modules/
vendor/bundle/
target/
dist/
build/
.next/
tmp/
log/
coverage/
```

Full rescan is available through `force: true` and deletes/rebuilds all rows for
the repo.

## 14. Code Scanning

### 14.1 MVP Language Coverage

Ship MVP support for:

```text
TypeScript / JavaScript
Ruby
Rust
Go
Python
```

Unsupported languages are indexed as files only. They can still contribute to
container/component generation through manifests and directories.

### 14.2 File Eligibility and Preview Limits

Scanner limits:

- Skip source parsing for files larger than 2 MiB and emit
  `scan.file_too_large`. They may still be indexed as files if their path is
  useful for generation.
- Treat files with NUL bytes in the first 8 KiB as binary, skip parsing, and
  emit `scan.binary_file_skipped`.
- Treat invalid UTF-8 in a source file as non-fatal: index the file path, skip
  symbol/import extraction for that file, and emit `scan.invalid_utf8`.
- Never send unbounded source text over IPC. `CodeRef.snippet` is capped at
  200 lines and 16 KiB after UTF-8 validation.
- Scanner warnings use `ValidationIssue` with `stage: "scan"` and are returned
  in `ScanSummary.warnings`.

### 14.3 Symbol Extraction

Extract these symbol kinds when the grammar supports them:

```text
class
module
interface
struct
enum
function
method
constant
```

Symbols are for jump-to-code and future L4 views. MVP generation should not
depend on perfect symbol extraction.

### 14.4 Import Extraction

Import extraction records:

- source file;
- target module string;
- resolved repo-relative target path when available;
- kind: `internal`, `external`, or `unknown`.

Resolution is best effort. Unresolved imports remain in the index for future
analysis but must not fail scans.

## 15. Generation

Generation emits `c4/model.generated.yml`. It never writes `c4/model.yml`.

### 15.1 Detection Rules

| Signal | Output | Notes |
|---|---|---|
| `package.json` | container | `tech` from package manager and deps; source roots `src`, `app`, `lib` |
| `Cargo.toml` | container | `tech: Rust`; source root `src` |
| `go.mod` | container | `tech: Go`; source root module dirs |
| `pyproject.toml` | container | `tech: Python`; source roots package dirs, `src` |
| `requirements.txt` | container | fallback Python container |
| `Gemfile` | container | `tech: Ruby`; Rails if `rails` dependency present |
| `Dockerfile` | container | if no manifest already covers same directory |
| `docker-compose.yml` service | container | one container per service |
| top-level source dir/package/module | component | within nearest container |
| internal imports | relationship | component or container level |
| external SDK/dependency | external system + relationship | generated and flagged |
| data-store driver | store container under generated/internal system + relationship | `kind: store` |

### 15.2 Generated Model Mapping

Generation always emits a schema-valid top-level model object.

Top-level generated fields:

- `name`: `GenerateOptions.systemName` when supplied; otherwise the authored
  effective model name when one exists; otherwise the titleized repo directory
  name.
- `description`: omit unless the generator can derive a useful one from package
  metadata.
- `generated: true`.

### 15.3 Generated System Mapping

Generation always emits or reuses one top-level internal system for repo-local
containers.

Selection rules:

1. If `GenerateOptions.systemSlug` is provided, use that slug.
2. Otherwise, if the authored effective model has exactly one internal system,
   use that system slug.
3. Otherwise, derive a slug from the repo directory name.

Name rules:

1. If `GenerateOptions.systemName` is provided, use that name.
2. Otherwise, if reusing an authored system, keep the authored name.
3. Otherwise, titleize the selected system slug.

The generated overlay includes this system with `generated: true`. If the same
slug is authored, authored system fields win during merge while generated child
containers remain available under the recursive child-merge rules. Every
generated container from repo-local manifests attaches under this system.
Generated external systems remain top-level siblings with `external: true`.

### 15.4 Generated Slug Collision Preflight

Before emitting `model.generated.yml`, generation builds a flattened slug table
from the authored effective model and from the candidate generated elements.
The candidate overlay must be globally slug-unique after this pass.

Rules:

- Same slug, same type, same parent slug: treat as the same intended element and
  allow authored fields to win during merge.
- Same slug with different type or different parent slug: treat as an ambiguous
  collision and choose a new generated slug by appending `_2`, `_3`, etc.
- Generated relationships must reference the final post-collision generated
  slugs.
- If a generated relationship would target an authored element that was not the
  intended same type/parent element, keep the relationship only after retargeting
  to the final generated slug.

This preflight prevents generation from making an otherwise valid authored
model fail with `semantic.duplicate_slug`.

### 15.5 Container Mapping

For each detected manifest:

1. Determine manifest directory.
2. Find nearest existing container for the same directory. If none, create one.
3. Slug is path-derived from the manifest directory; root manifest becomes app
   name if known, otherwise repo directory name.
4. Name is the package/module/service name when available, titleized slug
   otherwise.
5. `code` is the manifest directory or primary source root.
6. `generated: true`.

### 15.6 Component Mapping

For each container:

1. Identify source roots.
2. List first-level child packages/directories excluding generated/vendor/test
   directories.
3. Create one component per meaningful directory.
4. Slug is path-derived and globally unique.
5. Name is titleized directory/module name.
6. `code` is the directory path.
7. `generated: true`.

Default excluded component directories:

```text
test
tests
spec
specs
__tests__
fixtures
vendor
node_modules
target
dist
build
```

### 15.7 Relationship Inference

Internal imports:

- Map source file to nearest generated/authored component by `code` path or
  `element_sources`.
- Map resolved target file the same way.
- If both sides map and differ, generate relationship:

```yaml
- from: source_component_or_container
  to: target_component_or_container
  description: Imports
  generated: true
```

External dependencies:

- Manifest dependencies that match known SDK/client/data-store names generate an
  external system or store container and one relationship from the owning
  container.
- SDK/client dependencies generate top-level external systems with
  `external: true`.
- Data-store drivers generate `kind: store` containers under the generated or
  reused internal system, unless an authored store container with the same
  intended slug/type/parent already exists.
- Description format: `Uses <dependency>`.
- Relationship remains generated and should be visibly reviewable.

Generation must not infer runtime call graphs, dynamic dispatch, or message flow
from imports. Those are roadmap/LSP work.

| `sendgrid` | `sendgrid` | external system |
| `twilio` | `twilio` | external system |
| `sentry` | `sentry` | external system |

### 15.9 Overlay Output Header

Generated YAML begins with:

```yaml
# c4/model.generated.yml
# Regenerated by c4lens. Do not hand-edit; edit c4/model.yml instead.
# yaml-language-server: $schema=./schema.json
```

## 16. Watcher

The core watches:

- the repo root recursively, with the same ignore rules used by scanning;
- the `c4/` directory and its parent, so creating/deleting `c4/`,
  `model.yml`, `model.generated.yml`, or `schema.json` is observed;
- indexed source directories after a scan, including creation/deletion/rename
  events for files that were not previously indexed.

Behavior:

- Model file create/change/delete/rename events debounce for 150 ms, then
  validate and emit `model-changed` or `validation-failed`.
- Source file create/change/delete/rename events debounce for 500 ms, then
  incremental scan changed paths and emit `index-updated`.
- Atomic-save sequences such as create temp file, write, rename over original
  must coalesce into one model or source update.
- Watcher never writes `model.yml`.
- Watcher may update `model_cache`; it must not auto-regenerate
  `model.generated.yml` without an explicit user/CLI action in MVP.

## 17. CLI Contract

CLI binary name:

```text
c4lens
```

Commands:

```text
c4lens doctor [--repo PATH] [--json]
c4lens init [--repo PATH] [--name NAME] [--json]
c4lens validate [--repo PATH] [--json]
c4lens scan [--repo PATH] [--force] [--json]
c4lens generate [--repo PATH] [--scan] [--check] [--write] [--json]
c4lens schema [--repo PATH] [--json]
```

Defaults:

- `--repo` defaults to current working directory.
- `doctor` checks model/schema/generated-overlay presence, validation health,
  and setup recommendations without writing to the repository.
- `init` creates `c4/model.yml` and refreshes `c4/schema.json` without
  overwriting an existing authored model.
- `schema` refreshes `c4/schema.json` from the bundled schema.
- `validate` includes `model.generated.yml` if present.
- `generate` without `--write` prints or returns a diff but does not write.
- `generate --write` generates a candidate and applies it with `acceptAll: true`.
- `generate --write` creates `c4/`, `c4/schema.json`, and
  `c4/model.generated.yml` when needed, subject to the repo boundary and write
  lock rules. It never creates `c4/model.yml`.
- `generate --check` exits non-zero if generated overlay differs from disk.

Exit codes:

```text
0 success
1 validation errors or generate --check drift
2 invalid CLI usage
3 filesystem/repo error
4 scan/generation internal error
```

Human stdout should be concise. `--json` prints the corresponding typed payload
as JSON and writes human diagnostics to stderr.

## 18. Renderer UX Contract

### 18.1 App Shell

Primary layout:

- top app/menu region with repo name, validation state, generate action, search,
  export action;
- breadcrumb for C4 navigation;
- center xyflow canvas;
- right detail panel;
- optional bottom/status strip for scan/generation progress.

No landing page. Opening the app shows either the last repo, an open-folder
state, or the current repo model.

### 18.2 Navigator

Required interactions:

- Pan, zoom, fit view, minimap.
- Drill-down gesture: double-click a system node to open that system's
  container view; double-click a container node to open that container's
  component view.
- Breadcrumb up/across navigation.
- Select node to show detail panel.
- Detail panel action for drill-down: selected systems show `Open containers`
  and selected containers show `Open components`; each button matches the
  double-click destination.
- Breadcrumb/scope navigation is the canonical “up/back” path from container
  and component views.
- Hover/select dependency highlighting.
- Focus mode for incoming/outgoing dependencies.
- Deep-linkable in-webview route keyed by `(scope, selected)`.

Route shape:

```text
/#/context
/#/system/:systemSlug
/#/container/:containerSlug
/#/container/:containerSlug/component/:componentSlug
```

`/#/container/:containerSlug` renders the component view for that container.
`/#/container/:containerSlug/component/:componentSlug` renders the same view
with the component selected.

Route-to-scope mapping:

| Route | Derived view scope | Selection |
|---|---|---|
| `/#/context` | `{ level: "context" }` | none |
| `/#/system/:systemSlug` | `{ level: "container", slug: systemSlug }` | none |
| `/#/container/:containerSlug` | `{ level: "component", slug: containerSlug }` | none |
| `/#/container/:containerSlug/component/:componentSlug` | `{ level: "component", slug: containerSlug }` | `componentSlug` |

The `ViewScope.level` names describe the C4 view being rendered, not the slug
type carried in the route.

`/#/system/:systemSlug` and `/#/container/:containerSlug` entries implement the
scope transitions in 10.3 and 10.4 (system → container view, container →
component view).

Invalid route behavior:

- If the route slug does not exist or has the wrong type for the route, render a
  not-found state inside the canvas area and keep the repo/model loaded.
- If a component route names a component that is not a child of the route's
  container, render the container component view with no selection and show the
  component selection as not found.
- Loading the next valid route must not require a full model reload.

### 18.3 Detail Panel

For selected element, show:

- name, slug, type, status;
- description and tech;
- generated/authored provenance badge;
- incoming/outgoing relationships;
- code path if present or indexed;
- inline source preview when available;
- open in editor action.

If a non-external system or any container is selected, show a visible
drill-down control (keyboard-focusable, activatable with Enter/Space) that
navigates to:

- `Open containers`: system → container view
- `Open components`: container → component view

Generated elements and relationships must be visibly marked.

### 18.4 Generation Review

Flow:

1. User triggers Generate.
2. UI runs `scan_codebase` if needed.
3. UI runs `generate_model`.
4. UI presents diff summary and generated YAML diff.
5. User applies accepted changes.
6. UI refetches model after `model-changed`.

MVP may apply the full overlay only. If per-element apply is not implemented,
do not show per-element checkboxes.

### 18.5 Invalid Model State

If validation fails after a watched edit:

- keep rendering the last valid model;
- show validation errors with path and line/column when available;
- do not clear navigation state unless the selected element no longer exists
  after the next valid model loads.

## 19. Native and Security Constraints

Tauri capabilities must be scoped to:

- folder picker;
- reading the selected repo;
- writing `c4/model.generated.yml` and `c4/schema.json`;
- writing export files selected through save dialog;
- opening files through OS/editor integration.

The app must not:

- execute code from the opened repo;
- run package-manager scripts while scanning;
- send repo contents over the network by default;
- allow renderer-supplied paths outside the selected repo for code operations;
- allow arbitrary SQL from the renderer.

SQLite is owned by Rust core only. Do not use `tauri-plugin-sql` for renderer
SQL access.

## 22. Acceptance Criteria

### 22.1 Phase 0

- App launches on macOS.
- User can choose a folder.
- Renderer displays a hard-coded sample C4 model.
- xyflow pan/zoom/select works in WKWebView.
- ELK layout produces non-overlapping nodes for the sample.

### 22.2 Phase 1

- `c4lens validate` succeeds on a valid sample and fails on invalid YAML/schema
  and unresolved relationships.
- App loads `c4/model.yml` from selected repo.
- App validates through the Rust core and renders only valid effective models.
- Invalid watched edit shows errors and preserves last valid render.
- YAML anchors, aliases, merge keys, and duplicate keys produce deterministic
  parse errors.
- Model/control file symlinks that resolve outside the repo are rejected without
  modifying outside files.
- Invalid, escaping, and missing-inside-repo `code` paths are classified with
  distinct validation codes.
- Context, container, and component views derive from one model.
- Lower-level relationships aggregate upward correctly.
- Invalid deep links show a not-found state without crashing or reloading the
  model.
- SQLite index stores files, symbols, imports, and element source mappings.
- Incremental scan skips unchanged content, catches same-mtime/same-size content
  edits, and removes deleted files.
- Scanner limits skip oversized, binary, and invalid-UTF-8 files without
  emitting oversized snippets.
- Jump-to-code opens a local file and inline preview shows indexed snippet.

### 22.3 Phase 2

- `c4lens generate --write` creates deterministic `c4/model.generated.yml`,
  including from an empty repo with no existing `c4/` directory.
- Authored `model.yml` wins over generated overlay on slug conflicts.
- Generated relationships are additive and de-duplicated.
- UI generation flow shows scan progress, diff summary, and apply action.
- Concurrent GUI/CLI writers cannot corrupt the overlay, schema, SQLite index,
  or model cache; one writer succeeds or the other receives `repo.write_locked`.
- Generated elements/relationships show provenance badges.
- Search returns deterministic element and indexed-code results with bounded,
  literal query semantics.
- Dependency highlighting and focus mode work on all derived views.
- SVG, PDF, and PNG export write through native save dialog.
- Layout cache avoids recomputing unchanged scopes and invalidates when measured
  node dimensions or layout options change.
- macOS universal app artifact is produced.
